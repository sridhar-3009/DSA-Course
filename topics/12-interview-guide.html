Big-O & Interview Guide

Complexity analysis is the language of algorithm interviews. Understanding what Big-O means, how to derive it, and how to reason about space/time trade-offs is the final skill that ties everything together.

At a Glance

  • Big-O = upper bound — f(n) = O(g(n)) means f grows no faster than c*g(n) for large n. Drop constants and lower-order terms.
  • Big-Omega = lower bound — f(n) = Omega(g(n)). Big-Theta = tight bound (both O and Omega).
  • Amortized analysis — average cost per operation over a sequence. Dynamic array append is O(1) amortized despite occasional O(n) copies.
  • Space-time trade-off — using more memory often reduces time. Memoization: O(n) space, O(n) time vs O(1) space, O(2^n) time. Know when the trade-off is worth it.
Core Concepts

Drop Constants

O(2n) = O(n). O(n/2) = O(n). Big-O describes growth rate, not absolute speed. A O(n) algorithm with a high constant factor can be slower than O(n^2) for small n — but Big-O describes asymptotic behavior.

Add vs Multiply

Sequential loops: O(n) + O(m) = O(n+m). Nested loops over the same input: O(n) * O(n) = O(n^2). Nested over different inputs: O(n*m). This distinction catches many interview mistakes.

Recursion Complexity

Use the Master Theorem: T(n) = aT(n/b) + O(n^d). If d > log_b(a): O(n^d). If d = log_b(a): O(n^d log n). If d < log_b(a): O(n^log_b(a)). Merge sort: T(n)=2T(n/2)+O(n) → O(n log n).

Space Complexity

Count the extra memory used by the algorithm, not the input. Recursive DFS: O(h) stack space where h = tree height. In-place sort: O(1). Merge sort: O(n) auxiliary. Hash tables: O(n).

Algorithm Selection Guide

Use Two Pointers when

  • Array is sorted (or can be sorted)
  • Looking for a pair with a property
  • Removing duplicates in-place
  • Container with most water

Use Sliding Window when

  • Subarray / substring problem
  • Fixed or variable window size
  • Contiguous elements constraint
  • Max/min sum, longest substring

Use DP when

  • Overlapping subproblems
  • Optimal substructure
  • "Number of ways", min/max cost
  • Greedy fails or is unproven

Use BFS when

  • Shortest path (unweighted)
  • Level-order processing
  • All paths of same length
  • Minimum steps / hops
Interview Strategy
1

Clarify Constraints

Ask: What are n's bounds? Sorted? Distinct? Can I use O(n) space? These constraints often determine the algorithm. n=10^9 means O(n) is borderline; n=10^6 means O(n log n) is fine; n=100 means O(n^2) is acceptable.

2

Brute Force First

Always state the brute force O(n^2) or O(n^3) solution. This shows you understand the problem. Then optimize from there — interviewers want to see your thought process.

3

Identify the Pattern

Sorted array + pair sum = two pointers. Subarray max/sum = sliding window. Optimal substructure = DP. Shortest path = BFS/Dijkstra. Dependency ordering = topological sort. Matching brackets/parentheses = stack.

4

Code, then Walk Through

Write clean code with meaningful variable names. Then trace through a small example to verify. Check edge cases: empty input, single element, all duplicates, all same value.

5

State Complexity

Always state both time and space complexity with justification. "O(n log n) for the sort, O(n) for the hashmap, so O(n log n) total time and O(n) space." Interviewers expect this without being asked.

Interview Q&A

Common Interview Questions

O(g): upper bound. f(n) grows no faster than g(n). Most commonly used. Omega(g): lower bound. f(n) grows at least as fast as g(n). Theta(g): tight bound. f(n) grows exactly at the rate of g(n) — both O and Omega. In interviews, "O(n log n)" usually means Theta(n log n) informally. The information-theoretic lower bound for comparison sorting is Omega(n log n).
For divide-and-conquer T(n) = aT(n/b) + O(n^d): Case 1: d > log_b(a) → O(n^d). Work at top level dominates. Case 2: d = log_b(a) → O(n^d log n). Work balanced across levels. Case 3: d < log_b(a) → O(n^log_b(a)). Work at leaf level dominates. Examples: merge sort a=2, b=2, d=1 → log_b(a)=1=d → O(n log n). Binary search a=1, b=2, d=0 → log_2(1)=0=d → O(log n).
With n <= 10^4, O(n^2) is around 10^8 operations — borderline. With n <= 10^3, O(n^2) is easily acceptable. With n <= 100, even O(n^3) works. Rules of thumb: 10^8 operations per second on modern hardware. n=10^5: need O(n log n) or better. n=10^6: need O(n). n=10^9: need O(log n) or O(1). Always check constraints before coding.
Space = maximum stack depth * space per call. Binary search (recursive): O(log n) call stack depth * O(1) per frame = O(log n). Merge sort: O(log n) depth * O(n) auxiliary merge space = O(n). DFS on balanced tree: O(h) where h=log n. DFS on skewed tree: O(n). Fibonacci without memoization: O(n) stack depth despite O(2^n) total calls — only one path is on the stack at a time.
Best case: minimum over all inputs of size n. Quicksort on already-sorted input (with median pivot): O(n log n). Worst case: maximum over all inputs. Quicksort with always-last-element pivot on sorted input: O(n^2). Average case: expected over random inputs. Quicksort average: O(n log n). In interviews, state worst case unless told otherwise. Amortized is different — it is average cost per operation over a sequence, not over inputs.
